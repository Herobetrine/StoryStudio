# StoryStudio 路线图

本路线图从 0.7.0 的现状出发，把这个由 SillyTavern（酒馆）改造、拆分而来的独立项目，从可靠的单章生产与连续性内核推进为完整的长篇网文连载平台。当前酒馆兼容面是角色卡、世界书和 Prompt preset 的资源导入，不包含通用插件运行时。

版本号代表能力边界，不代表发布日期。每个阶段只有在对应验收标准有运行证据后才进入稳定发布。

## 路线原则

1. **权威状态优先。** 新能力通过 proposal、证据、版本检查和原子采纳进入正式数据。
2. **作者体验优先。** 默认界面展示创作语义；hash、revision、Artifact 等工程细节进入高级模式。
3. **来源可追溯。** 摘要、记忆、图谱、质量问题和 Retcon 影响都能跳回原始章节或采纳记录。
4. **本地优先。** 无模型时仍可管理、检索、审校、备份和发布；联网调用保持显式。
5. **兼容不等于执行任意代码。** SillyTavern 资源继续兼容；扩展能力通过版本化 SDK 和权限边界提供。
6. **阶段可验收。** 每项能力同时交付数据合同、迁移、API/UI、测试、恢复说明和公开文档。

## 0.8 — 统一章节生产主线

### 目标

把“AI 候选、策划 Copilot、Workflow、质量、版本”整合为作者可以从今天的任务一路完成到正式稿的单一生产链。

### 0.8.1 今日工作台

当前进度：Dashboard V1 的只读投影、`GET /api/story-studio/projects/:projectId/dashboard` 和顶层“今日”UI 已在 0.7 后续工作树中完成，覆盖作品进度、焦点章节、确定性下一步、今日工作项、章纲过期、伏笔债务、近期章节、Story State 摘要、刷新/重试和桌面/移动响应式布局。工作项已能跳到对应章节、卷、账本伏笔视图或 Workflow；0.8 继续补齐 Workflow/Copilot/Quality 聚合、写作统计、日目标、存稿与可恢复活动。

交付：

- 当前作品、当前卷、当前章和今日字数目标；
- 存稿量、连续更新天数和最近写作记录；
- 系统计算的下一步：补设定、补章纲、选择方向、继续 run、处理质量问题或发布；
- 待审批 Artifact、冲突、过期章纲、伏笔到期和质量 blocker 聚合；
- 一键进入下一章或恢复最近一次 Workflow。

验收：

1. 同一权威状态始终计算出确定的下一步和原因；
2. 刷新或重启后工作台恢复同一任务；
3. 待办项可以跳到对应作品、章节、Workflow 步骤或正文位置；
4. 至少覆盖空项目、写作中、待审批、过期、完成和冲突六类状态。

### 0.8.2 Copilot → Workflow 交接

当前进度：核心交接合同已在 0.7 后续工作树中完成。ready 方向卡可以创建 Workflow V2 run；服务端只接受 `sessionId/artifactId/optionId`，重新验证 authority 与候选血缘，把完整方向集合和每个方向的 3–12 节点 `sourceEventChain` 转为 `brainstorm-direction`，只批准所选项并直接进入 `plan`。Plan 必须用 `sourceEventCoverage` 逐项映射合法 beat，Draft 再消费映射。handoff V2 只持久化来源坐标与摘要；GET 列表/详情保持只读，恢复只由认证 POST 触发，标准路径是相同 start POST 重放，且不重新调用 Provider。同命令幂等、部分物化恢复、stale/跨章/活动 run 阻断已有合同测试。0.8 继续补充 stale 差异解释、完整浏览器链路和更作者化的交接反馈。

交付：

- Copilot 方向卡增加“用此方向开始章节流程”；
- 新 run 固定绑定 session、attempt、direction、证据和权威摘要；
- 方向转换为类型化 brainstorm/plan 输入，避免复制粘贴；
- 完整 `sourceEventChain` 与 Plan `sourceEventCoverage` 进入可审计血缘；
- Copilot session 变 stale 时展示差异并要求刷新或显式重做。

验收：

1. 一个已完成方向可以一次操作创建 V2 run；
2. run 中的方向、证据和来源摘要与 Copilot 完全一致；
3. 重复点击返回同一幂等结果；
4. 项目或章节版本变化时旧方向不会进入新 run；
5. 3–12 个源事件无截断进入 Plan，每个源顺序恰好映射一次且只引用已声明 beat；
6. GET 不改变中断 run，认证 start POST 重放恢复同一 run 且不增加 Provider 调用；
7. 端到端测试覆盖“策划 → 选方向 → 计划 → 正文 → 审查 → 采纳”。

### 0.8.3 Workflow 快速模式

交付：

- 默认使用“构思、计划、初稿、审校、定稿”五段作者视图；
- 内部 12 步保持原合同，在高级模式显示；
- 默认隐藏 hash、revision、原始 Artifact JSON；
- 每个阶段显示输入、模型动作、作者决定、下一步和可恢复状态；
- 支持预设“快速成章”“严格审校”“仅人工写作”流程体验。

验收：

1. 新用户无需编辑 JSON 即可完成完整章节；
2. 快速模式与高级模式操作同一个 run，不产生双轨状态；
3. 每个高层阶段都能追溯到底层步骤和 Artifact；
4. 崩溃恢复、取消、幂等重试和 writer 租约合同保持通过。

### 0.8.4 作者级 ChangeSet 审批

交付：

- 把原始 ChangeSet JSON 投影为“新增事实、状态变化、人物知识、时间推进、关系变化、伏笔动作”；
- 每条变更显示来源原文、理由、目标实体和冲突；
- 支持逐条接受、排除或编辑后形成新的 proposal；
- 采纳前展示正文与账本的联合摘要。

验收：

1. 普通审批路径不展示原始 JSON；
2. 所有可视化操作都编译回严格 ChangeSet Schema；
3. 逐条排除不会破坏引用完整性；
4. 采纳仍保持正文与 Story State 原子提交。

### 0.8.5 Character Card 与人物实体绑定

交付：

- Character 资源可绑定一个 Story State 人物实体；
- 支持别名匹配、人工确认、解除和重绑；
- Prompt 使用卡片描写，连续性使用权威人物 ID；
- 卡片与人物实体字段分歧显示为 proposal，而不是静默覆盖。

验收：

1. 一个激活角色卡只能有一个有效人物绑定；
2. 重名不会自动合并；
3. 导入、导出、重命名和资源删除保持引用完整；
4. 上下文预览同时显示资源来源与权威人物 ID。

### 0.8.6 工程基线

当前进度：Windows/Ubuntu Node 20 CI、Provider 手工重定向与响应大小边界、npm `files` 发布白名单、发布文档链接/绝对路径门禁、package dry-run、真实浏览器冒烟、覆盖率报告，以及架构/安全/贡献/路线图/变更记录已经进入 0.7 后续工作树。正式 release workflow 仍待完成。

交付：

- Windows / Ubuntu Node 20 CI；
- Provider 重定向和响应大小边界；
- npm 发布白名单；
- 发布包内 Markdown 链接与可移植路径检查；
- 仓库内真实浏览器冒烟测试；
- 覆盖率报告和最低门槛；
- 公开架构、安全、贡献、路线图和变更记录。

验收：

1. pull request 必须通过语法、全量测试、质量门禁、生产依赖审计和浏览器冒烟；
2. Git 跟踪树和 npm 包清单均不含 `.research/`、`.private/`、内部验收记录、`data/`、`logs/`、测试私稿或凭据；
3. Provider 重定向、错误体、JSON、SSE/NDJSON 和最终正文超限测试全部通过；
4. 桌面与 375×812 视口覆盖启动、编辑、策划、Workflow、质量和导出主路径。

## 新增优先队列

以下八项面向长篇网文的真实生产链，优先于“再加一个泛化聊天入口”。它们仍按后续版本交付，不是
0.7.0 已实现能力。

| 优先级 | 能力 | 建议版本 | 核心产物 |
| --- | --- | --- | --- |
| P0 | 酒馆原生 JSON 资源导出 | 0.8 | Character/World Info/Preset 原生导出、往返 fixture、损失报告 |
| P0 | 稳定锚点批注与创作决策日志 | 0.8 | 跨修订锚点、orphan 处理、决策取代链 |
| P0 | 人物 Voice Bible 与术语 Bible | 0.9 | 声音约束、称谓矩阵、术语正字表与来源例句 |
| P0 | 类型化连续性状态包 | 0.9 | 物品、伤势、能力冷却、金钱、衣着伪装、行程、势力资源 |
| P1 | Story Tests | 0.9 | 作者声明的作品断言、证据化失败和采纳/发布门禁 |
| P1 | 研究资料库与引用图 | 0.9/1.0 | source/excerpt/claim/正文或设定锚点血缘 |
| P1 | Reader Room | 1.0/1.1 | 不可变盲读包、结构化反馈、版本对照和 A/B 分组 |
| P1 | 作品包装工作台 | 1.0 | 书名、简介、标签、封面 brief、黄金三章 A/B |

### 原生资源导出

- 导出 Character Card V2/V3 JSON、World Info / Character Book JSON 和受支持 Prompt preset；
- 每次转换列出保留、默认化、StoryStudio-only 和不可表达字段；
- PNG 原始像素尚未持久化时只承诺 JSON，不声称 PNG 字节级无损往返；
- 原生资源导出与 `.story-studio.json` 便携内容副本使用不同入口和格式标识。

验收：固定 fixture 完成 `SillyTavern -> StoryStudio -> SillyTavern JSON` 语义等价测试；任何损失
都进入机器可读报告，不静默丢弃。

### 稳定锚点、Voice/术语与类型化状态

- 锚点绑定章节/修订、UTF-16 范围、quote、前后文/段落指纹和内容摘要；无法唯一重定位时
  标记 orphan；
- 创作决策保存理由、备选、证据、状态和 `supersedes`，可关联 Beat、Story State、质量 issue
  与原文；
- Voice Bible 保存角色句式、词汇域、礼貌度、称谓、禁用表达和有来源的正/反例；
- 术语 Bible 保存 canonical form、别名、错误形式、允许出现范围和来源；
- 类型化状态第一批覆盖物品、伤势、能力/冷却、金钱、衣着/伪装、行程和势力资源，所有变化
  继续通过 proposal、人工门和原子采纳。

验收：插句/删段/移段后锚点结果确定且歧义时 fail-closed；声音与术语漂移能定位逐字证据；
库存、冷却、金额、伤势、位置和伪装不变量进入固定测试。

### Story Tests、研究资料、Reader Room 与作品包装

- Story Tests 使用白名单 DSL 声明知识时点、资源守恒、能力冷却、伏笔期限和术语揭示等作品
  断言，在生成前、采纳前、Retcon 后和发布前运行；
- 研究资料库区分外部来源、摘录、claim 与作品 Canon，保留作者、日期、页码/时间码、许可备注
  和引用图；
- Reader Room 从精确项目版本生成盲读包，收集掉点、困惑、期待、情绪、人物好感和自由评论，
  反馈只形成报告/proposal；
- 作品包装把书名、简介、标签、封面 brief 和黄金三章 A/B 作为版本化资产，并检查营销承诺
  是否有正文事实支撑。

验收：Story Test 失败给出实际值、期望值和来源；研究 claim 可反查摘录；旧稿读者反馈不会冒充
新稿结果；包装版本绑定项目版本，A/B 报告区分观察、推断与作者决定。

## 0.9 — 长篇结构、图谱与跨章质量

### 目标

让 StoryStudio 从“知道当前状态”升级为“理解人物弧、剧情线、场景和长期债务”，并把这些信息以作者可读的图表和跨章审查呈现。

### 0.9.1 人物弧模型

交付：

- 角色欲望、恐惧、误区、价值观、秘密、能力限制；
- 人物弧阶段、触发事件、关键选择、代价和阶段结论；
- 关系阶段与人物弧联动；
- 章节对人物弧的推进/停滞/倒退标记。

验收：

1. 人物弧条目有稳定 ID、来源、有效时间和状态；
2. 同一章节的 proposal 可以同时更新人物弧与关系；
3. 生成上下文只注入当前时点有效阶段；
4. 人物弧变更可追溯到正文证据和采纳记录。

### 0.9.2 剧情线、场景与 Beat

交付：

- 主线、支线、人物线和谜题线；
- 线目标、阶段、阻塞、汇流、完成条件；
- 章节内场景和 Beat：目标、冲突、转折、信息释放、钩子；
- Beat → 分段写作 → 累积上下文 → 整章合并。

验收：

1. 每章可关联多个剧情线并声明贡献；
2. Beat 顺序、人物在场和时间地点通过校验；
3. 分段生成不会丢失已批准 Beat 与累积上下文；
4. 合并正文保留段落来源和候选血缘。

### 0.9.3 伏笔债务系统

交付：

- 埋设、强化、误导、提醒、兑现、废弃六类动作；
- 计划兑现窗口、最晚章节、当前热度和逾期状态；
- 按卷、剧情线、人物和紧急度筛选的债务看板；
- Workflow/章纲生成前提示本章应触及的伏笔。

验收：

1. 每个伏笔动作绑定来源章节和证据范围；
2. 时间倒退、提前泄露和重复兑现会被阻断或告警；
3. 到期规则确定性可测试；
4. 看板与生成上下文读取同一权威数据。

### 0.9.4 连续性可视化

交付：

- 可缩放时间线；
- 人物/势力/地点/物品关系图；
- POV × 事实知识矩阵；
- 事件、事实、记忆和章节的来源图；
- 图中选中节点可跳回原文、章纲和采纳记录。

验收：

1. 图表完全由权威状态派生，可删除重建；
2. 每条边都有来源和有效时间；
3. 图表过滤遵守 POV 和未来信息边界；
4. 1,000 章项目在性能预算内完成增量更新与交互。

### 0.9.5 Retcon 影响分析

交付：

- 修改事实、人物状态、时间、关系或章节顺序前计算影响；
- 列出依赖章节、知识边、记忆、伏笔、章纲、检索块和质量报告；
- 生成可审批的修复计划；
- Retcon 后只重建受影响派生数据。

验收：

1. 影响结果包含逐字来源证据；
2. 不确定依赖显式标记，不静默忽略；
3. 未确认分析不会改写权威状态；
4. 应用 Retcon 时使用版本检查和原子事务；
5. 测试覆盖事实替代、删章、移章、时间变化和人物合并。

### 0.9.6 跨章质量与内联修复

交付：

- 跨章重复、节奏、称呼、POV、时间、人物声音和设定漂移；
- 正文内高亮、问题队列和逐项处理；
- 段落级 before/after diff；
- 局部改写 proposal 和人工采纳；
- 作品、卷和规则维度的质量趋势。

验收：

1. 每个问题定位到章节、段落和原文范围；
2. 局部修复只修改批准范围；
3. diff、建议、采纳和恢复均有血缘；
4. 跨章检查支持增量运行；
5. 回归集包含误报和跨章一致性固定用例。

## 1.0 — 稳定恢复、发布与规模化

### 目标

形成可长期连载、可恢复、可发布、可迁移的稳定版，并对数据兼容和扩展合同作出明确承诺。

### 1.0.1 恢复中心

交付：

- 删除章回收站与保留期限；
- 章节逐行 diff；
- 整次 Workflow 的联合恢复计划；
- 明确区分“导入后生成新 ID 的便携内容副本”和“保持项目 ID 与全部 sidecar 的完整备份”；
- 项目、候选、Workflow、Copilot、Quality、检索和版本历史的完整备份清单；
- 停服快照创建、校验、演练和恢复向导。

验收：

1. 恢复前自动验证 manifest、摘要、版本和依赖；
2. run 级恢复覆盖正文、章纲、Story State 和 sidecar；
3. 恢复操作自身生成新的审计记录；
4. 随机中断测试证明不会留下混合状态；
5. 便携内容副本导入后只建立新项目血缘，不挂接旧 sidecar；
6. 保持身份的完整备份可在全新目录恢复原 ID 并通过完整性检查。

### 1.0.2 发布中心

交付：

- TXT、Markdown、DOCX、EPUB 导出；
- 单章、选卷、全书和增量发布；
- 平台标题、正文、作者的话与格式模板；
- 发布前敏感格式、缺章、重复章、字数和质量检查；
- 发布包清单、摘要和版本记录。

验收：

1. 同一权威版本重复导出得到确定性内容摘要；
2. 输出顺序严格匹配卷章目录；
3. 中文标点、空行、标题和编码在目标格式中通过快照测试；
4. 导出失败不改变作品发布状态；
5. 发布记录可以定位到精确项目版本和章节修订。

### 1.0.3 连载排期与存稿

交付：

- 日更计划、目标字数、存稿量和预计耗尽时间；
- 章节发布状态与计划发布时间；
- 中断、补更、加更和卷末缓冲；
- 今日任务与发布中心联动。

验收：

1. 排期使用明确时区；
2. 移章、删章和插章后计划可预测地重算；
3. 已发布章节不会被普通编辑静默替换；
4. 提醒和统计都能追溯到权威发布记录。

### 1.0.4 存储与性能

交付：

- 流式导入导出，降低大项目内存峰值；
- sidecar 分页、配额、归档与级联清理；
- 检索重建 singleflight 和并发队列；
- 覆盖所有写路径的跨进程租约/CAS，或明确升级为单 writer 存储服务；
- 1,000/3,000 章基准与长时间运行测试。

验收：

1. 接近容量上限的导入导出峰值在公开预算内；
2. 两个进程竞争同一数据目录时只有一个写入者成功；
3. sidecar 达到配额后给出可操作的归档提示；
4. 任务取消、超时和进程退出后没有永久占用租约；
5. 性能基准纳入 release 证据。

### 1.0.5 稳定兼容承诺

交付：

- Schema、Prompt Profile、Workflow、备份和扩展合同的版本策略；
- 迁移支持窗口；
- release notes、升级检查和回滚说明；
- 安全更新与弃用策略。

验收：

1. 所有持久格式都有版本、迁移器和固定 fixture；
2. 旧版数据升级和备份恢复进入 CI；
3. 破坏性变化经过主版本或明确迁移；
4. 公开文档与实际 API/Schema 一致。

## 1.1 — 小说扩展生态与智能运营

### 目标

在 1.0 稳定数据与恢复边界之上开放小说专用 Extension SDK，并增加模型路由、成本分析、幕后仿真和作品复盘。

### 1.1.1 Extension SDK V1

交付：

- 版本化 manifest；
- 能力权限：上下文读取、proposal 提交、项目存储、UI 插槽、任务调度；
- 安装、升级、停用、卸载和数据迁移生命周期；
- 有版本的事件总线；
- 项目级隔离存储、配额和审计；
- 示例扩展与开发者工具。

安全与执行前置：

- V1 默认采用声明式扩展包；UI 由宿主渲染声明式组件，或运行在 opaque-origin
  sandbox iframe，第三方同源脚本不作为权限边界；
- 每个安装实例拥有独立 principal、项目授权和可撤销 grant，所有能力由服务端再次校验；
- 后端扩展若开放，运行在独立进程，通过版本化 RPC 白名单访问宿主；
- 网络、模型调用、密钥、文件系统、后台任务分别授权并计入配额；
- 任务调度复用持久 Job Center，具备 checkpoint、重启恢复、取消、并发和费用限制。

V1 核心合同：

- `AuthoringContextSnapshotV1`：不可变、版本绑定、按 POV 和授权范围过滤的作者上下文；
- `ContextContributionV1`：声明 slot、优先级、token 预算、来源证据、可见性和 freshness，
  并进入最终 Messages / Injection Ledger；
- `ExtensionProposalV1`：携带 authority fingerprint、证据、幂等键和基线版本，只通过现有
  proposal / ChangeSet / 原子采纳路径写入；
- `ExtensionEventEnvelopeV1`：声明事件 ID、类型、Schema、因果 ID、权威版本和 payload digest，
  固定重复投递、顺序、重放、游标、超时与背压语义；
- `ExtensionJobV1`：接入统一 Job Center，而不是创建仅存在于进程内的扩展任务表。

首批贡献面：

- `uiContributions`：今日工作台卡片、章纲侧栏、上下文来源面板和只读图表；
- `ruleContributions`：版本化确定性质量规则与发布前检查；
- `actionContributions`：命令面板、工作台按钮、Workflow 模板和带 dry-run 的 Creator Actions；
- `contextContributions`：分层记忆、摘要、检索建议和世界书推荐；
- `artifactProducers`：只生成类型化 Artifact / proposal 的分析器。

验收：

1. 扩展只能使用 manifest 声明并获准的能力；
2. 扩展写入先形成 proposal，再进入人工审批和权威事务；
3. 扩展崩溃不会破坏主应用数据；
4. 安装包具备摘要、来源和兼容版本；
5. 升级失败可以回到旧版本；
6. 示例扩展在 Windows、macOS、Linux 上通过合同测试；
7. 扩展页面直接调用宿主 API 时仍受 principal/grant 和项目边界约束；
8. stale proposal、重复事件、乱序重放、慢订阅者和扩展重启均有固定回归；
9. 扩展升级迁移失败可回滚，卸载数据保留/删除策略明确；
10. CPU、内存、存储、网络、模型费用和取消配额均有负面测试。

### 1.1.2 SillyTavern 有界适配器

当前状态：只有 Character Card、World Info / Character Book 和 Prompt preset 的资源导入兼容；
没有 `SillyTavern.getContext()`、`eventSource / event_types`、`extension_settings`、Slash Command、
Quick Reply、STscript、`/api/plugins/*`、JavaScript/CSS 注入或 TavernHelper 插件运行时。
本条路线不会把“资源可导入”写成“插件可直接安装运行”。

交付：

- 继续维护角色卡、世界书和 Prompt 资源兼容；
- 与前述原生 JSON 导出共享往返 fixture 和损失报告；
- 定义独立版本的 `st-bounded-v1` 兼容 Profile；
- 将 `getContext()` 映射为只读 `AuthoringContextSnapshotV1`；
- 将 `extension_settings` 映射为扩展全局/项目命名空间存储；
- 将 `eventSource.on()` 映射为白名单宿主事件，将 `eventSource.emit()` 限定在扩展自有命名空间；
- 生成兼容性报告，以 `native / bridged / migratable / unsupported` 标记每个依赖接口；
- 对 DOM hook、任意 JavaScript/CSS 注入、生成拦截器、服务器插件路由、Function Tool 和内部模块
  import 给出明确迁移结果；
- 提供迁移指南，把聊天型插件机制转换为小说 proposal/事件。

验收：

1. 适配器拥有独立版本和测试矩阵；
2. 任何适配接口都不能绕过 StoryStudio 权威写入；
3. 不支持的接口在安装前给出结构化诊断；
4. 直接复制酒馆插件仍不会绕过 manifest/权限/兼容性检查；
5. 插件测试 fixture 固定到明确提交摘要，不跟随公开仓库 HEAD 漂移；
6. 至少完成三条互补迁移样例：
   - 分层记忆扩展：来源绑定、stale 传播和上下文贡献；
   - 连续性状态扩展：人物、时间、物品和关系变化形成 Story State proposal；
   - 世界书推荐扩展：检索与模型重排后形成资源激活建议。

### 1.1.3 模型路由与成本

交付：

- 按任务绑定模型、上下文窗口、价格、速度和结构化输出能力；
- 草稿、审校、摘要、重排使用不同路由；
- 预算上限、失败回退和本地模型优先策略；
- token、耗时、失败率、采纳率和质量变化趋势。

验收：

1. 每次调用记录任务、模型、参数、token、耗时和结果摘要；
2. 密钥与正文不会进入遥测；
3. 预算超限在调用前提示；
4. 回退不会静默改变权威任务合同；
5. 用户可以按作品导出或删除统计。

### 1.1.4 幕后仿真与提案

交付：

- NPC、势力和环境的幕后状态推进；
- 仿真只生成 proposal，不直接改写正式时间线；
- 与剧情线、伏笔和人物知识边联动；
- 对多个未来方向做代价、冲突和一致性比较。

验收：

1. 仿真状态与权威状态物理隔离；
2. 每个提案列出假设、因果链和来源；
3. 采纳前执行 Retcon/连续性影响分析；
4. 同一随机种子与输入可复现实验记录。

### 1.1.5 作品复盘

交付：

- 字数、更新、质量、生成成本、候选采纳率和返工率；
- 人物/剧情线/伏笔活跃度；
- 卷级节奏与读者反馈导入；
- 可导出的阶段复盘报告。

验收：

1. 每项指标有公开定义；
2. 指标可追溯到版本与记录；
3. 删除统计不会影响作品权威内容；
4. 报告区分事实、推断和作者备注。

## 路线图外延候选

当前主路线尚未完全展开的高价值增量包括：

1. 崩溃可恢复的浏览器草稿日志；
2. 可暂停的开书与前 30 章启动向导；
3. 保存最终逐字 Messages 与来源范围的 Generation / Injection Ledger；
4. 来源绑定的分层记忆与双时态 Canon；
5. 系列级共享 Bible、作者可编辑 Beat 与叙事指纹；
6. 酒馆原生 JSON 导出、稳定锚点批注、Voice/术语 Bible 和类型化状态包；
7. Story Tests、研究资料库、Reader Room 和作品包装；
8. 网文读者承诺/回报账本、持久 Job Center 和项目 Fork/Compare/Merge。

这些条目继续遵守本路线的权威状态、proposal、人工审批、版本检查与原子采纳原则。实施时应把数据合同和验收条件并入相应版本，而不是建立平行状态系统。

## 跨版本持续门禁

每个阶段都应持续满足：

- `npm ci`
- `npm run check`
- `npm run docs:check`
- `npm test`
- `npm run quality:check`
- `npm audit --omit=dev`
- 仓库内浏览器 E2E
- 数据迁移与恢复固定用例
- npm 打包内容检查
- 公开文档路径与链接检查

任何新 AI 能力还应证明：

1. 调用前上下文来源可见；
2. 输出先进入候选或 proposal；
3. 模型没有权威写入权限；
4. 采纳使用精确版本与引用校验；
5. 失败、取消、重试和恢复都有稳定状态；
6. 作者可以回看“模型看到了什么、建议了什么、最终采纳了什么”。
